# v3.5 Refactor — Progress & Tracking Log

## READ THIS FIRST

This file tracks the v3.5 refactor: five "no-regret" Python refactors of the
audio core. Background and reasoning are in the scoping plan at
`~/.claude/plans/we-re-scoping-a-rosy-raccoon.md` — not in the repo.

The work was triggered by a Rust-port re-scoping in 2026-05-02. The Rust
port itself is **not** in scope for v3.5; these refactors stand on their
own merits and (separately) would be the prerequisites if Rust ever
revisited.

## Current status: IN PROGRESS — started 2026-05-02

## The five refactors (in execution order)

### A. Push sinks behind per-sink queues, drain post-tick
**Goal:** No sink call inside the bus tick blocks the tick. Pattern already
exists for PCM/MP3/SDR rebroadcast (`bus_manager.py:1081-1101`).

**Sinks to convert (from `_deliver_audio` at `bus_manager.py:683-898`):**

After auditing each sink target, the actual situation is much narrower than the original list suggested. Most sinks are already off-tick by design — wrapping them in another queue is busywork.

- [x] **Broadcastify** (`stream_output.send_audio`) — converted 2026-05-02. Synchronous ffmpeg stdin write; needed converting + encoder lock fix.
- [x] **Automation recorder** (`automation_engine.recorder.feed`) — converted 2026-05-02. Synchronous lame stdin write, single writer (no peer-thread audit issue).
- [x] **EchoLink** (`echolink_source.send_audio`) — converted 2026-05-02. Non-blocking pipe write with flush; already had drop-on-full semantics (BlockingIOError caught), single writer.
- [x] **Mumble TX** (`mumble.sound_output.add_sound`) — converted 2026-05-02. pymumble's `SoundOutput.add_sound` is internally thread-safe (uses `self.lock` around `self.pcm` mutation; encoder thread also locks). Frame-buffer state (was `self._mumble_buf` on the tick side) moved into the drain thread (`self._mumble_drain_buf`) so only one thread mutates it.
- [—] **Transcriber** (`transcriber.feed`) — *already* off-tick. `feed()` is a thin enqueue onto `stream.feed_queue` with a per-stream worker thread (`_stream_worker`) doing the heavy work. Wrapping it in another queue is no benefit.
- [—] **Speaker** (`gw._speaker_enqueue`) — *already* enqueue-only; drained by PortAudio callback.
- [—] **Remote audio TX** (`remote_audio_server.send_audio`) — non-blocking socket send, breaks out on `BlockingIOError`. Already designed not to block. Could convert for total isolation but low priority.
- [—] **Link TX** (`link_server.send_audio_to`) — `_send_to` puts onto a per-endpoint `_send_queue` non-blocking with drop-on-full. *Already* off-tick.
- [—] **Loop recorder** (`loop_recorder.feed`) — `LoopSegment.feed` puts to `_write_queue` (drained by writer thread). Segment rotation closes old segments in a background thread. *Already* off-tick (was fixed earlier in v3.x for an 800ms stall).

**Acceptance:** `_tick_trace` shows no sink-attributable >5ms ticks during
a 10-min listen test. Per-sink drain threads visible in `htop`.

### B. Vectorize the noise gate
**Goal:** Replace per-sample Python loop in `_apply_noise_gate()`
(`audio_util.py:916-953`) with a fast inner loop.

**Status: done 2026-05-03.** The IIR has feedback that switches its
coefficient per sample (attack vs release), which doesn't vectorise
cleanly in pure numpy. Settled on numba-jit (`@njit`, `cache=True`) for
the inner loop — already a transitive dep via moonshine, so no new
install requirement. Benchmarks on a 2400-sample (50 ms) chunk:

- Cold first call (compile + run): ~550 ms (one-shot, JIT cache on disk)
- Warm: **0.011 ms/call** versus old loop estimated at 12-24 ms/call

That's >1000× steady-state. Output is **bit-identical** to the old loop
verified on a synthetic 2400-sample transient signal — same threshold
formula, same envelope coefficients, same soft-knee `(env/thr)^2`
below threshold. Pure-Python fallback present for any environment
where numba happens to be missing (no behaviour change, just slower).

### C. Snapshot routing/config at tick start
**Goal:** One read of `gw.config.*`, `gw._muted_sinks`, `gw._sink_gains`,
`gw.link_endpoints` etc. per tick into a `TickContext`. Pass through.

**Status: done 2026-05-03.** `TickContext` is a `@dataclasses.dataclass(frozen=True, slots=True)` with the read-side gateway state: sink target object refs (mumble/stream_output/transcriber/remote_audio_server/loop_recorder/automation_engine/echolink_source/link_server/web_config_server/th9800_plugin/kv4p_plugin), routing snapshot (frozenset(muted_sinks), dict copies of sink_gains/link_endpoints/link_endpoint_settings), config flags actually read on the tick (audio_rate/enable_vad/radio_to_echolink), and the stream_trace ref. Built once per tick at the top of `_tick_loop` via `_build_tick_context()`. Passed into `_deliver_audio(output, bus_id, ctx)` and `_handle_listen_tick(output, chunk_size, ctx)`. Mid-tick config races on those snapshotted fields are now impossible by construction.

**Out of scope for C (moves in D):** mutating writes to `gw.*` (level meters, `_link_tx_levels`, `_link_ptt_active`) still target gw directly. `gw.calculate_audio_level(audio)` is still called as a method on gw because it has internal state. v3.5-D pulls those out of gw into a returned tick-result struct.

### D. Separate audio-critical state from UI state
**Goal:** Bus tick returns level meters / duck flags / PTT state as a
result struct. One consumer writes them back to `gateway_core` after the
tick.

**Status: scoped subset done 2026-05-03.** The genuinely tick-internal
read-modify-write fields — the level meters that smooth via
`int(prev * 0.7 + new * 0.3)` — are now owned by `BusManager._meters`
(dict for the 5 sink-level meters) and `BusManager._link_tx_meters`
(per-endpoint dict). The tick reads/writes those locally, never
re-fetching from gw mid-tick. After the bus loop completes, a single
post-tick mirror copies the values to `gw.stream_audio_level`,
`gw.mumble_tx_level`, etc. and back-fills `gw._link_tx_levels` so
existing UI consumers (status JSON, level bars, link endpoint /level
field) keep working.

**Out of scope (deferred):** pure write-only fields (`gw.sdr_ducked`,
`gw.audio_capture_active`, `gw._last_mixer_trace_state`,
`gw.last_audio_capture_time`) still write to gw directly — they have
no read-modify-write loop, so isolating them is mostly a unit-test
concern. `gw._link_ptt_active` is shared with web_server / MCP and
isn't safe to relocate without auditing every reader.

**Acceptance bar (unit test):** not yet written — the tick still has
external side effects (the off-tick sink enqueues hit BusManager
state, transcription writes to its own queues, etc.) so a "pure
function" unit test would be inventing scaffolding for marginal
benefit. Skipping for now; level-meter isolation alone is the
practical win.

### E. Make all sources push instead of pull
**Goal:** Replace `bus.tick()` calling `source.get_audio(chunk_size)`
per source with sources pushing into per-source queues from their reader
threads. `bus.tick()` drains from queues only.

**E.1 — instrumentation done 2026-05-03.** Before converting any source,
added per-source `get_audio()` timing in `audio_bus._timed_get_audio`
(wraps every call site in ListenBus / SoloBus / DuplexRepeater /
SimplexRepeater). Counters: calls, no_audio (returned None), total_ms,
max_ms, avg_ms, idle_s. Surfaced via `BusManager.get_source_stats()`,
`/sourcestats` HTTP, and the `bus_source_stats` MCP tool. Same pattern
as the v3.5-A++ sink instrumentation. Lets us pick which sources
actually need converting based on evidence rather than theory.

**E.2 — conversions to follow.** Watch `bus_source_stats` for sources
with `max_ms` > tens of ms or non-trivial `avg_ms × calls_per_sec`.
Those are the ones worth pushing into reader threads. Cheap sources
stay pull.

**Sources to consider:**
- [ ] KV4P (Opus decode + resample currently inside `get_audio`)
- [ ] File playback (decode-on-demand)
- [ ] Web mic
- [ ] Mumble RX
- [ ] EchoLink RX
- [ ] (SDR and LoopPlayback already async internally — just expose push)

**Acceptance for E.2:** No converted source's `get_audio()` does work —
all decode/resample happens in a reader thread. Removing the polling
pattern lets the tick sleep predictably.

## Notes / decisions / gotchas

- **Drop-oldest policy on sink queue overflow.** `deque(maxlen=8)` means a slow consumer loses old chunks rather than ballooning latency. Audio gap > runaway latency for every sink we currently route. If a future sink wants different semantics (e.g. lossless), give it its own queue type.
- **Level meters stay on the tick.** Off-tick delivery doesn't change UI metering — `gw.stream_audio_level` etc. are still set on the bus tick using the pre-call connection state. UI feel unchanged.
- **stream_trace timing for converted sinks now records 'enq', not ms.** The actual call no longer happens on the tick, so per-tick ms is misleading. If we ever need drain-side timing diagnostics, add a separate trace point inside `_do_sink_send`.
- **Off-tick delivery exposes pre-existing thread races on the sink side.** When `broadcastify` was converted, `StreamOutputSource._encoder.stdin` was being written by both the new drain thread (real PCM) and the existing `Broadcastify-keepalive` thread (silence). Concurrent writes to `subprocess.Popen.stdin` interleave bytes mid-sample → MP3 output decodes as static / "silence". Pre-change the writes serialised naturally because send_audio ran on the bus tick at predictable cadence, keeping `_last_audio_time` fresh enough that keepalive never fired alongside it. Fixed by adding `_encoder_lock` around both write sites in `audio_sources.py`. **Lesson for the rest of v3.5-A:** before converting each sink, audit its existing object for any other thread that touches the same downstream resource (subprocess stdin, socket, file handle) and lock if needed.
- **Don't read too much into the broadcastify Icecast bitrate.** ~31 kbps is libmp3lame's floor for 48 kHz mono regardless of the requested `-b:a 16k`. The encoder silently rounds up. So `darkice_stats.send_rate` is *not* a useful signal for "is the encoder being double-fed". Verify regressions by listening to the stream, not by watching the bitrate.
- **Reload kills drain threads silently.** `BusManager.reload()` calls `stop()` then `start()`. `stop()` joins all `SinkDrain-*` threads. `start()` does not recreate them — it spins up the bus tick thread only. Lazy init in `_enqueue_sink` only fires when the dict entry is `None`; after stop() the dict still held the old (now-dead) Thread objects, so the next enqueue appended to a deque nothing was draining and audio dropped. Fixed by clearing `_sink_queues / _sink_events / _sink_threads` at the end of `stop()`. Routing reloads now correctly recycle drain threads. **Lesson:** any state that the drain threads hold has to be reset together with the threads themselves.
- **Mumble TX needs a sink-side VAD when wired off a non-listen bus.** The pre-existing `self._listen_vad_pass` is only computed for the listen bus and only gated *listen-bus* mumble delivery. When the user routes mumble off a solo bus (e.g. `aioc → th9800 → mumble`), the AIOC mic captures the radio's speaker output continuously — so even with the radio squelched there's a low-level signal flowing, and pre-v3.5-A had no gate either. Fixed by adding `BusManager._mumble_sink_vad(audio)` (own envelope state, mirrors `gateway_core.check_vad`) used by the non-listen-bus branch. Broadcastify intentionally has no equivalent gate — that sink *needs* a continuous stream to keep Icecast from disconnecting (handled by `Broadcastify-keepalive` filling silence between real frames).
