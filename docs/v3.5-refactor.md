## The five refactors (in execution order)

### A. Push sinks behind per-sink queues, drain post-tick
**Goal:** No sink call inside the bus tick blocks the tick. Pattern already
exists for PCM/MP3/SDR rebroadcast (`bus_manager.py:1081-1101`).

**Sinks to convert (from `_deliver_audio` at `bus_manager.py:683-898`):**

After auditing each sink target, the actual situation is much narrower than the original list suggested. Most sinks are already off-tick by design — wrapping them in another queue is busywork.

- [x] **Broadcastify** (`stream_output.send_audio`) — converted 2026-05-02. Synchronous ffmpeg stdin write; needed converting + encoder lock fix.
- [x] **Automation recorder** (`automation_engine.recorder.feed`) — converted 2026-05-02. Synchronous lame stdin write, single writer (no peer-thread audit issue).
- [x] **EchoLink** (`echolink_source.send_audio`) — converted 2026-05-02. Non-blocking pipe write with flush; already had drop-on-full semantics (BlockingIOError caught), single writer.
- [x] **Mumble TX** (`mumble.sound_output.add_sound`) — converted 2026-05-02. pymumble's `SoundOutput.add_sound` is internally thread-safe (uses `self.lock` around `self.pcm` mutation; encoder thread also locks). Frame-buffer state (was `self._mumble_buf` on the tick side) moved into the drain thread (`self._mumble_drain_buf`) so only one thread mutates it.
- [—] **Transcriber** (`transcriber.feed`) — *already* off-tick. `feed()` is a thin enqueue onto `stream.feed_queue` with a per-stream worker thread (`_stream_worker`) doing the heavy work. Wrapping it in another queue is no benefit.
- [—] **Speaker** (`gw._speaker_enqueue`) — *already* enqueue-only; drained by PortAudio callback.
- [—] **Remote audio TX** (`remote_audio_server.send_audio`) — non-blocking socket send, breaks out on `BlockingIOError`. Already designed not to block. Could convert for total isolation but low priority.
- [—] **Link TX** (`link_server.send_audio_to`) — `_send_to` puts onto a per-endpoint `_send_queue` non-blocking with drop-on-full. *Already* off-tick.
- [—] **Loop recorder** (`loop_recorder.feed`) — `LoopSegment.feed` puts to `_write_queue` (drained by writer thread). Segment rotation closes old segments in a background thread. *Already* off-tick (was fixed earlier in v3.x for an 800ms stall).

**Acceptance:** `_tick_trace` shows no sink-attributable >5ms ticks during
a 10-min listen test. Per-sink drain threads visible in `htop`.

### B. Vectorize the noise gate
**Goal:** Replace per-sample Python loop in `_apply_noise_gate()`
(`audio_util.py:916-953`) with numpy envelope-follower.

**Acceptance:** Gate produces bit-identical output to current implementation
on a recorded test fixture (or audibly identical on a live test). Per-tick
gate time drops 5-10× on the bus that uses it.

### C. Snapshot routing/config at tick start
**Goal:** One read of `gw.config.*`, `gw._muted_sinks`, `gw._sink_gains`,
`gw.link_endpoints` etc. per tick into a `TickContext`. Pass through.

**Acceptance:** Bus tick code reads only from the snapshot, never from
`gw.*` directly. Mid-tick config races impossible by construction.

### D. Separate audio-critical state from UI state
**Goal:** Bus tick returns level meters / duck flags / PTT state as a
result struct. One consumer writes them back to `gateway_core` after the
tick.

**Acceptance:** First-ever unit test of `bus.tick()` — synthesize input
PCMs, assert output PCM + result struct. If the test is hard to write,
this refactor isn't done.

### E. Make all sources push instead of pull
**Goal:** Replace `bus.tick()` calling `source.get_audio(chunk_size)`
per source with sources pushing into per-source queues from their reader
threads. `bus.tick()` drains from queues only.

**Sources to convert:**
- [ ] KV4P (Opus decode + resample currently inside `get_audio`)
- [ ] File playback (decode-on-demand)
- [ ] Web mic
- [ ] Mumble RX
- [ ] EchoLink RX
- [ ] (SDR and LoopPlayback already async internally — just expose push)

**Acceptance:** No source's `get_audio()` does work — all decode/resample
happens in a reader thread. Removing the polling pattern lets the tick
sleep predictably.

## Notes / decisions / gotchas

- **Drop-oldest policy on sink queue overflow.** `deque(maxlen=8)` means a slow consumer loses old chunks rather than ballooning latency. Audio gap > runaway latency for every sink we currently route. If a future sink wants different semantics (e.g. lossless), give it its own queue type.
- **Level meters stay on the tick.** Off-tick delivery doesn't change UI metering — `gw.stream_audio_level` etc. are still set on the bus tick using the pre-call connection state. UI feel unchanged.
- **stream_trace timing for converted sinks now records 'enq', not ms.** The actual call no longer happens on the tick, so per-tick ms is misleading. If we ever need drain-side timing diagnostics, add a separate trace point inside `_do_sink_send`.
- **Off-tick delivery exposes pre-existing thread races on the sink side.** When `broadcastify` was converted, `StreamOutputSource._encoder.stdin` was being written by both the new drain thread (real PCM) and the existing `Broadcastify-keepalive` thread (silence). Concurrent writes to `subprocess.Popen.stdin` interleave bytes mid-sample → MP3 output decodes as static / "silence". Pre-change the writes serialised naturally because send_audio ran on the bus tick at predictable cadence, keeping `_last_audio_time` fresh enough that keepalive never fired alongside it. Fixed by adding `_encoder_lock` around both write sites in `audio_sources.py`. **Lesson for the rest of v3.5-A:** before converting each sink, audit its existing object for any other thread that touches the same downstream resource (subprocess stdin, socket, file handle) and lock if needed.
- **Don't read too much into the broadcastify Icecast bitrate.** ~31 kbps is libmp3lame's floor for 48 kHz mono regardless of the requested `-b:a 16k`. The encoder silently rounds up. So `darkice_stats.send_rate` is *not* a useful signal for "is the encoder being double-fed". Verify regressions by listening to the stream, not by watching the bitrate.
